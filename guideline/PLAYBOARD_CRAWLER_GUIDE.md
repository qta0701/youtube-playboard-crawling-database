# 플레이보드 크롤러 기술 가이드라인 (PLAYBOARD_CRAWLER_GUIDE.md)

이 문서는 **플레이보드 크롤러 탭**에서 작동하는 Selenium 기반 데이터 수집 엔진의 구동 메커니즘, 안정성 확보 정책 및 로그인 관리 가이드를 설명합니다.

---

## 1. 🔑 구글 로그인 세션 유지 메커니즘 (중요)

### 1.1. 동작 원리 및 세션 이식성
* **로컬 프로필 경로 고정**: 크롤러는 시스템 드라이브와 프로젝트 위치에 구애받지 않도록, Windows 사용자 홈 폴더 내에 크롬 사용자 프로필 데이터를 강제로 리다이렉션하여 격리 저장합니다.
  - **결정 경로**: `C:\Users\<Windows사용자명>\.adsense_auto_workflow\chrome_profiles\playboard`
* **최초 1회 로그인 후 유지**:
  - `login_mode=True` 옵션을 켜고 크롤러를 실행하면, 셀레늄이 실행하는 크롬 브라우저가 사용자 홈 디렉터리의 로컬 프로필 폴더를 마운트합니다.
  - 브라우저 창이 뜨면 구글 계정으로 **최초 1회 로그인**을 완료합니다.
  - 로그인 세션 정보와 쿠키는 사용자 홈 디렉터리의 크롬 프로필 하위의 Local Storage와 Cookie 데이터베이스에 영구 기록됩니다.
  - **따라서 이후 크롤러를 재실행할 때는 별도로 로그인 절차를 거치지 않아도 자동으로 로그인 세션이 유지되어 안정적으로 수집이 진행됩니다.**

### 1.2. 🛡️ 구글 로그인 차단 우회 메커니즘 (중요)
* **구글 계정 차단 우회 (바이너리 서명 패치 및 순정 브라우저 모사)**: 
  - 구글은 셀레늄 표준 ChromeDriver의 고유 서명(예: `cdc_adoQpoasnfa76pfcZLmcfl_` 변수 명칭 등) 및 브라우저 우회 도구(`selenium-stealth` 등)가 남기는 미세한 JS 모킹 흔적(languages, webgl 등의 조작 흔적)을 감지하여 구글 로그인 시 "안전하지 않은 브라우저" 오류를 표출하고 원천 차단합니다.
  - 이 문제를 해결하기 위해, 크롤러 기동 직전 ChromeDriver 실행 파일의 바이너리 코드 내부에서 `cdc_` 식별자 문자열을 임의의 문자열로 자동 치환하는 **바이너리 수준 패치**를 적용해 자동화 흔적을 원천 제거합니다.
  - 또한, 주입 흔적을 역추적당해 구글 로그인 차단을 부르는 `selenium-stealth` 마스킹 라이브러리 실행을 완전히 제외함으로써 순정 크롬 브라우저와 100% 동일한 개체 지문(Fingerprint)을 발송하도록 조치했습니다.
  - 이를 통해 플레이보드 로그인 페이지(`https://playboard.co/account/signin`)에서 구글 로그인을 다이렉트로 진행하여 안전하게 자동 차단을 우회하고 성공시킬 수 있습니다.
  - 로그인을 성공적으로 마친 후 대시보드에서 **'수동 재개'** 버튼을 클릭하면 세션 검증이 완료되고, 플레이보드 연동 로그인이 즉시 성공하여 크롤링 단계로 매끄럽게 전환됩니다.

## 2. 🏗️ 크롤러 핵심 구동 로직 및 백그라운드/최소화 최적화

### 2.1. 🤖 봇 탐지 우회 엔진 개편 (Standard Selenium + ChromeDriver 바이너리 패치)
* **`webdriver-manager` 자동 버전 매칭**: 기존 `undetected-chromedriver`의 빈번한 Chrome 버전 업데이트 불일치 오류를 막기 위해, 공식 `webdriver-manager`를 최우선으로 구동하여 PC에 설치된 Chrome 버전에 맞는 최적의 ChromeDriver를 자동 설치합니다.
* **ChromeDriver 바이너리 자동 서명 패치**: 크롤러 기동 시 사용될 ChromeDriver.exe 파일의 바이너리 데이터를 실시간으로 읽어 구글 계정 보안 탐지에 걸리는 `cdc_` 고유 식별 키를 동일한 크기의 더미 텍스트로 자동 치환하여 디스크에 덮어씁니다.
* **`selenium-stealth` 비활성화 및 순정 지문 사용**: `selenium-stealth`가 수행하는 WebGL/Navigator 객체 조작 지문은 최신 구글 로그인 엔진에 봇 마커로 역탐지되어 차단을 유발하므로, stealth 라이브러리를 비활성화하여 순정 브라우저의 지문을 그대로 사용하고 대신 바이너리 패치만으로 구글 로그인을 완벽히 해결합니다.
* **GUI 모드 실행 원칙 (`headless=False`)**: 보안 감지(Cloudflare 등) 회피 및 로그인 세션 입력을 원활히 처리하기 위해 기본 구동을 `headless=False` 상태로 설정하는 것을 원칙으로 합니다.
* **랜덤 딜레이 (`random_delay`)**: 단순 고정 딜레이를 지양하고, 사람과 유사한 불규칙한 동작 모사를 위해 페이지 로딩, 스크롤 동작 전후에 `random_delay(min_val, max_val)`를 필수 호출하도록 설계되었습니다.

### 2.2. 💤 백그라운드 최소화(비활성화) 구동 보장 및 Throttling 해제 (오류 로그 차단 포함)
크롤러 동작 중에 사용자가 크롬 브라우저를 최소화(Minimize)하거나 다른 문서/인터넷 서핑 작업을 할 때, 크롬 브라우저가 강제로 CPU 사용률과 JS 타이머를 지연시키는 Throttling 정책을 무력화하고, 불필요한 GPU/DevTools 경고가 터미널에 출력되지 않도록 다음과 같은 세부 특수 옵션을 도입했습니다.
* `--disable-background-timer-throttling`: 백그라운드 탭 타이머 제한 정책 해제
* `--disable-backgrounding-occluded-windows`: 브라우저 창이 다른 화면에 가려져 있어도 비활성 상태로 전환되는 것을 차단
* `--disable-renderer-backgrounding`: 렌더러 프로세스 우선순위 격하 방지
* `--disable-features=CalculatePageVisibility`: 크롬 탭의 가시성 계산 연산을 강제 정지하여 상시 '활성(Visible)' 상태로 인지시킴
* `--disable-features=IntensiveWakeUpThrottling`: 백그라운드 대기 탭의 알람/타이머 강제 지연 해제
* `--disable-features=ThrottleDelayableQueueOnCpuUsage`: CPU 부하에 따른 대기열 지연 정책 강제 해제
* `--log-level=3` / `--silent` / `excludeSwitches=['enable-logging']`: DevTools 리스닝 정보 및 ui direct_composition overlay 등 GPU 관련 콘솔 에러 출력을 완벽히 억제하여 터미널 창 가독성 유지

### 2.3. 🔕 포커싱 강제 팝업 간섭 차단 및 병렬 안전성 강화 (포트 & 락)
* **강제 팝업 차단**: 기존 버전에서는 백그라운드로 전환될 때 화면을 활성화하기 위해 주기적으로 `window.focus()`와 `switch_to.window`를 트리거하여 화면을 맨 앞으로 강제로 띄워 올렸습니다. 이는 크롤링 도중 사용자가 다른 작업을 할 때 큰 불편(간섭)을 야기했습니다. 개편된 버전에서는 강력한 Throttling 차단 크롬 옵션 덕분에 화면을 팝업할 필요가 전혀 없으므로, **강제 포커싱 로직을 완전히 소거**했습니다. 이에 따라 사용자는 크롬 창을 최소화해둔 상태에서도 지연 없이 완전한 정상 속도로 백그라운드에서 크롤링이 조용히 동작하게 됩니다.
* **원격 디버깅 포트 랜덤 할당 (9300~9500)**: 병렬로 여러 워커가 기동될 때 크롬 디버깅 포트 충돌로 인한 브라우저 기동 거부 현상을 막기 위해, 드라이버 기동 옵션에 `--remote-debugging-port`를 9300~9500 사이의 랜덤 포트로 할당합니다.
* **초기화 락 (Driver Lock) 적용**: `ChromeDriverManager().install()` 등이 다중 스레드에서 동시에 실행될 경우 로컬 드라이버 캐시 파일 접근 충돌이 발생합니다. 이를 스레드 락(`PlayboardCrawler._driver_init_lock = threading.Lock()`)으로 감싸 순차적으로 드라이버가 빌드 및 기동되도록 보호합니다.

### 2.4. 📊 사용자 지정 수집 개수 및 순위(Rank) 기반 스크롤 판단
* **수집 개수 파라미터 (Limit) 연동**: 대시보드 옵션 패널에서 수집하고 싶은 데이터의 한도(Limit, 기본값 100)를 사용자가 직접 설정해 전달할 수 있습니다.
* **`div.current` 순위값 정밀 역추적**: 테이블 행(tr) 개수로만 목표량을 판단하던 단순 방식에서 탈피하여, 플레이보드의 순위 영역인 `<div class="current">순위숫자</div>` 요소의 텍스트를 실시간 파싱 및 변환합니다. 수집된 랭킹의 최댓값이 사용자가 입력한 목표 순위에 완벽하게 도달할 때까지 1부터 순차적으로 로드 완료된 것을 검증한 후 스크롤을 중단합니다.

### 2.5. 🔑 로그인 상태 요소 기반 정밀 검증 및 수동 로그인 무한 대기
* **로그인 요소 기반 검증**: 구글 로그인 페이지 링크(`a[href*="/account/signin"]`, 텍스트 '로그인')가 노출되고 있다면 미로그인 상태로 진단하여 즉시 사용자에게 윈도우 알림 및 Speech Off 효과음을 발송합니다. 로그인 버튼이 사라지거나 프로필 이미지 요소(`div.menu--signed` 또는 `div.menu__picture` 등)가 확인되면 로그인 완료로 진단합니다.
* **수동 재개 무한 대기**: 기존의 120초 자동 만료 후 진행 방식을 폐지하고, 사용자가 로그인을 완료한 후 대시보드에서 **'수동 재개'** 버튼을 클릭하여 검증이 완료될 때까지 크롤러는 브라우저를 띄운 채 무한히 대기합니다.

### 2.6. 📡 비로그인 기본 100개 지원 및 로그인 팝업 실시간 감지
* **비로그인 수집 한도 완화**: 로그인 모드를 켜지 않더라도 기본적으로 100개(사용자가 입력한 Limit)까지 수집을 허용하도록 필터를 수정했습니다.
* **로그인 요구 팝업 감지 알림**: 비로그인 모드로 순회 도중 플레이보드의 로그인 요구 팝업(Login Wall)이 화면상에 검출되는 즉시, 사용자에게 윈도우 OS 네이티브 알림 메시지와 경고음을 전송하고 크롤러 스레드는 스크롤 루프를 안전하게 조기 탈출(Break)하여 현재까지 수집된 데이터를 안정적으로 보존합니다.

### 2.7. '전체' 카테고리 vs '일괄 크롤링' 모드 비교
🛠️ 수집 옵션 설정 단계에서 동작하는 두 옵션은 수집 범위와 실행 프로세스 면에서 다음과 같이 확연히 다릅니다.

1. **카테고리 (Category)를 "전체"로 지정한 경우 (단일 수집)**:
   - **대상**: 플레이보드 사이트에서 각 분야의 경계를 두지 않은 **종합 랭킹 차트(all)** 하나만 조회합니다.
   - **동작**: 단 1개의 URL(예: `.../shorts-ranking/all`)에 접속하여 상위 랭킹(비로그인 20개 / 로그인 최대 100개)만 빠르게 수집하고 저장합니다.
2. **"📦 전체 카테고리 일괄 크롤링" 체크박스를 활성화한 경우 (일괄 순회 수집)**:
   - **대상**: 종합 '전체' 카테고리를 포함한 **모든 카테고리**(전체 및 세부 서브 카테고리 - 음악, 게임, 스포츠, 영화 등)들을 대상으로 합니다.
   - **동작**: 백그라운드 루프를 통해 각 카테고리별 URL들을 하나씩 순차적으로 접속하여 크롤러를 재기동합니다. 수집이 완료되면 카테고리별로 개별 CSV 파일(`shorts_batch_전체`, `shorts_batch_동물` 등)을 생성함과 동시에, 최종적으로 모든 데이터를 하나로 병합한 대규모 통합 CSV 파일(`ALL`)을 생성하여 저장합니다.
   - *주의*: 수십 개의 페이지를 순회하므로 단일 수집 대비 수집 시간이 길어집니다.
   - *통합 중복 처리*: 병합된 통합 CSV(`ALL`) 및 대시보드 `"All"` 탭의 경우, 개별 카테고리 통합본과의 100% 중복을 유발하는 `"전체"` 카테고리 수집본은 데이터에서 제외하여 중복 영상 노출을 방지합니다. 사용자 선택 탭에서 `"전체"`를 직접 클릭하면 `"전체"` 카테고리 단독 데이터가 정상 조회됩니다.

## 3. 📜 Lazy Loading 대응 최적화 스크롤 기법

Playboard는 화면 하단으로 내릴수록 데이터가 추가로 바인딩되는 Lazy Loading 방식을 사용하므로 이를 격파하기 위해 고도화된 스크롤 모션을 실행합니다.

### 3.1. Element-Based Stepped Scrolling (원소 기반 계단식 스크롤)
- 픽셀 단위로 스크롤을 내리면 렌더링 속도와 네트워크 지연 때문에 데이터를 인식하지 못하는 경우가 빈번합니다.
- 이에 따라 크롤러는 현재 DOM에 로딩된 마지막 행 요소(`tr.chart__row`)를 수집해 이를 뷰포트 중앙으로 끌어올리는 `scrollIntoView({block: 'center'})` 명령을 자바스크립트로 직접 전송합니다.
- ActionChains API를 거치지 않고 순수 JS로 브라우저 콘솔을 직접 제어하므로 스크롤 응답성이 크게 향상됩니다.

### 3.2. Wiggle Scrolling, OS 레벨 강제 복원/포커싱, 스크롤 최하단 자동 스킵 및 사용자 알림 연동 (스로틀링 격파)
- 백그라운드 구동 시 윈도우 OS 커널 및 크롬 자체의 스로틀링(Throttling) 정책으로 인해 렌더링이 일시 지연되어 3회 연속 데이터 변화가 감지되지 않는 정체 상태(동일 수량 4회 반복 노출)가 될 경우, 다단계의 지연 해제 장치가 자동 가동됩니다.
- **스크롤 최하단(바닥) 도달 시 자동 스킵 (알림 전 탈출)**:
  - 랭킹 총 수량이 100개 미만으로 부족하여 화면 하단 끝에 도달했을 때 불필요한 수동 개입 알림창이 뜨는 것을 막기 위해, 스크롤 후 수집된 개수에 변화가 없을 때 즉시 자바스크립트로 스크롤 최하단(바닥) 도달 여부(`(window.innerHeight + window.scrollY) >= (document.documentElement.scrollHeight - 50)`)를 감지합니다.
  - 스크롤 최하단(바닥)에 도달한 상태이면서 **2회 연속** 데이터 개수 변화가 없다면, 진짜 페이지의 끝에 다다른 것으로 간주하여 사용자 알림 및 복원 대기 루프를 생략하고 즉시 스크롤 루프를 종료(`break`)하여 다음 카테고리 수집으로 자동 전환합니다.
- **ctypes Win32 API 기반 강제 복원 및 활성화**:
  - 스크롤 하단이 아니면서 백그라운드 지연으로 인해 수집 정체가 **3회** 연속 지속될 경우, `PlayboardCrawler._force_focus_os_window()` 메서드를 호출하여 Windows OS의 `user32.dll` API를 이용해 크롬 창의 HWND 핸들을 획득합니다.
  - 해당 창이 작업 표시줄로 최소화되어 있다면 `ShowWindow(hwnd, SW_RESTORE)`를 호출해 **원래의 크기와 위치를 100% 완벽히 유지한 상태로** 화면에 즉각 복원(De-minimize)합니다.
  - 이어서 `SetForegroundWindow(hwnd)` 및 `SetActiveWindow(hwnd)`를 호출하여 브라우저를 OS 상의 최상위 활성 창으로 강제 활성화하고 백그라운드 대기 상태를 해제합니다.
- **Wiggle Scrolling**:
  - 활성화 자극과 병행하여 화면을 순간적으로 위로 150px 이동시켰다가 아래로 150px 재이동시키는 스크롤 모션(Wiggle)을 실행하여 크롬 렌더러가 신규 리소스 로딩을 재개하도록 강제합니다. (기존의 불안정하고 다른 링크 오클릭을 유발하던 body 클릭 자극은 오작동을 차단하기 위해 완전히 제거되었습니다.)
- **지속 정체 시 사용자 수동 개입 유도 및 무한 대기**:
  - 자동 창 복원(`_force_focus_os_window`)을 시도한 직후 즉시 사용자 수동 개입 유도를 위한 무한 대기 모드로 진입합니다.
  - 이 시점에 즉시 경고 효과음(`play_notification_sound`)과 함께 **"[수집 대기] 자동화 브라우저 창을 활성화해 주세요"** 타이틀의 윈도우 OS 알림(`show_notification`)을 사용자에게 발송합니다.
  - 알림 발송 직후 크롤러는 대기 루프에 진입하여, Javascript `document.hasFocus()`를 통해 브라우저 창이 실제 활성 상태(포커스 획득)가 될 때까지 안전하게 무한히 대기(Pause)합니다.
  - 브라우저 창 활성화가 감지되면, 추가 로드를 적극 유도하기 위해 주기적인 자동 Wiggle 스크롤(2초 간격)을 시도하며, 실제 추가 요소(`tr.chart__row`)가 화면 상에 새롭게 로드(증가)될 때까지 대기 상태를 유지합니다. (대기 중에도 대시보드를 통한 프로세스 중단은 보장됩니다.)
  - 추가 요소 로드가 최종 확인되면 자동으로 대기가 즉시 해제되며 크롤링 동작을 재개합니다.
- **F5(새로고침) 재시도 및 수동 재개 정책**:
  - **F5(새로고침) 재시도**: 브라우저 창이 활성화되었으나 15초가 경과해도 추가 로드를 수행하지 못할 때(스크롤의 끝에 도달한 경우), 크롤러는 브라우저를 **F5(새로고침)** 하고 기존에 설정된 랭킹 정렬 기준 탭을 다시 클릭한 뒤, 데이터 파싱 행 목록을 리셋하여 처음부터 다시 수집을 시도합니다.
  - **수동 재개 시 재수집**: 사용자가 대시보드에서 **'수동 재개'** 버튼을 클릭하면, 수집 대기 상태가 즉각 해제되고 해당 카테고리의 브라우저를 새로고침(F5) 및 탭 재설정 후 처음부터 완전히 다시 수집을 기동합니다.
  - **다음 카테고리 스킵**: 대기 중 사용자가 **'다음 카테고리'** 버튼을 클릭하면, 현재 카테고리 수집 루프를 즉각 탈출하고 다음 순서의 카테고리 수집으로 매끄럽게 전환됩니다.

### 3.3. 로그인 벽(Login Wall) 조기 탐지
- 비로그인 모드로 100개 이상의 수집을 요청하거나 대량 수집 중 로그인 유도 모달이 노출되면, 스크롤 루프가 영구히 대기하는 리소스 낭비가 발생합니다.
- `_check_login_wall()` 함수를 통해 화면 상의 로그인 팝업, 로그인 문장 등의 XPath를 매 턴 검출하여 로그인 벽이 감지되면 즉시 안정적으로 대기를 탈출하도록 예외 처리를 수행했습니다.

## 4. 🎟 수집 알림 및 효과음 연동
* **로그인 대기 알림**: 100개 이상의 랭킹 행을 수집하기 위해 사용자의 세션 로그인이 필요한 상황 시, 백그라운드 환경에 있는 사용자를 즉각 호출합니다. 윈도우 OS의 네이티브 알림(`show_notification`) 창을 띄우고 알림음(`play_notification_sound`)을 동시에 송출하여 청각적으로 유도합니다.
* **크롤링 완료 알림**: 모든 크롤링 수집 프로세스가 마무리되고 브라우저가 안전하게 종료될 때(`Chrome WebDriver closed successfully.`), 사용자에게 수집 완료 효과음(`play_sound`)과 함께 성공 완료 네이티브 알림(`show_notification`)을 전송하여 즉각적인 기동 마무리를 피드백합니다.
* **크롤링 예외 중지 알림**: 수집 처리 도중 예상치 못한 스크립트 예외(TypeError, 봇 탐지 차단 등)가 발생하면, 수집 동작을 즉각적이고 물리적으로 전면 정지(`break`)시킵니다. 이후 즉시 사용자에게 경고 효과음(`play_notification_sound`) 및 "유튜브 크롤러 기동 에러 발생" 타이틀의 윈도우 OS 토스트 알림을 전송하여 사용자가 크롤러 정지 상태를 인지할 수 있도록 유도합니다.
* **수집 지속 정체 시 사용자 활성화 알림**: 스크롤 수량이 3회 이상 전혀 변하지 않는 백그라운드 스로틀링 정체 현상이 5회 및 8회 연속 지속되는 한계 상황 봉착 시, 사용자에게 경고음 및 "자동화 브라우저 창을 활성화해 주세요" 윈도우 알림을 전송하여 수동으로 브라우저 탭을 전면 활성화하도록 유도합니다.

## 5. 📂 로그 파일 자동 정제 및 디버그 로깅 강화

### 5.1. 최신 5개 로그 파일만 유지 (`cleanup_old_logs`)
* 무분별하게 로그 파일이 누적되어 하드디스크 공간을 낭비하는 현상을 방지하기 위해, 앱이 재실행(기동)될 때마다 가장 최근의 로그 파일 **5개만 남기고** 오래된 로그 파일들은 자동으로 영구 삭제 처리합니다.
* 파일의 수정 일자(`getmtime`)를 동적으로 정렬 및 탐색하여 백그라운드에서 안전하게 소거합니다.

### 5.2. 디버그 로그의 실시간 CMD 및 로그 파일 기록
* 셀레늄 브라우저 드라이버 로드 실패(1, 2, 3차 단계별 에러), 스크롤 에러, 웹 요소 미발견 등의 상황 발생 시 기존의 debug 수준을 넘어 **warning** 및 **error** 레벨로 로깅의 중요도를 올렸습니다.
* 또한, 로거의 콘솔 핸들러 작동 레벨`DEBUG` 수준으로 지정하여, 크롤링 동작 중 발생하는 모든 핵심 상세 내부 프로세스가 사용자의 명령 프롬프트(CMD) 화면에 실시간으로 세밀하게 인쇄됩니다.
* **콘솔(CMD) 동기화 복원**: `app.py` 기동 시 `setup_logger('crawler')`를 통해 로깅 설정을 명시적으로 수행하여 백그라운드 터미널(명령 프롬프트) 창에도 수집 관련 디버그 로그가 실시간 출력되도록 개선했습니다.
* **현황판 로그 세션 보존**: Streamlit 페이지가 중단/재개 동작에 의해 리렌더링될 때 실시간 로그 현황판 위젯의 내용이 초기화되는 현상을 막기 위해, 로그 스트림 메시지들을 `st.session_state['log_history']` 리스트 버퍼에 지속 누적(최대 100개)하여 보존합니다. 오직 새로운 **🚀 크롤링 시작** 클릭 시에만 현황판 로그 버퍼가 리셋됩니다.
* **CSV 저장 성공 로깅 정책**: 크롤링이 완료되어 로컬 디스크에 파일로 써질 때, 단일 크롤링의 저장부와 일괄 크롤링의 개별/통합(`ALL`) 저장부 직후에 파일의 **절대 경로** 및 **파일명**을 콘솔과 로그 파일에 `logger.info`로 상세히 인쇄합니다. 이를 통해 디바이스 환경의 확실한 디스크 파일 갱신 및 경로 위치를 쉽게 검증할 수 있습니다.
* **불필요한 디버그 로그 차단 정책 (Clean Terminal)**: 데이터베이스 다량 적재 시 매 비디오 ID가 저장될 때마다 출력되던 대량의 `Inserted new short: <video_id>` 로그(`logger.debug`)를 비활성화(주석 처리)하여, 터미널이 의미 없는 로그로 채워지는 현상을 방지하고 로그 파일의 부피를 대폭 절약하였습니다.
* 이를 통해 문제 발생 시 콘솔 로그와 로그 파일을 대조하여 즉시 원인을 디버깅할 수 있습니다.

## 6. ⏯️ 수동 재개 및 프로세스 중단 제어 아키텍처

### 6.1. 비동기 시그널링 설계 (`st.session_state` 연동)
Streamlit의 동기식 스크립트 실행 환경에서 실시간 프로세스 제어를 보장하기 위해 크롤러 세션에 시그널링 플래그를 이식했습니다.
* **사용 가능한 제어 신호**:
  - `stop_requested`: 프로세스 및 브라우저 강제 종료 요구 플래그
  - `resume_requested`: 수동으로 해당 카테고리의 크롤링을 F5 새로고침 후 처음부터 재수집하도록 요구하는 플래그
  - `skip_requested`: 현재 수집 중인 카테고리를 건너뛰고 즉시 다음 카테고리로 넘어가도록 요구하는 플래그
* **실시간 인터랙티브 단추 (4열 구성)**:
  - **🚀 크롤링 시작**: 플래그들을 모두 초기화하고, `PlayboardCrawler` 인스턴스를 메모리(`st.session_state['crawler_instance']`)에 등록한 후 실행을 개시합니다.
  - **⏯️ 수동 재개**: 수동 로그인 대기 또는 수집 정체 대기 상태를 해제하고, 해당 카테고리를 새로고침(F5)하여 처음부터 다시 수집을 시작합니다.
  - **⏭️ 다음 카테고리**: 현재 카테고리 수집 루프를 즉시 건너뛰고(Skip) 다음 카테고리 수집으로 매끄럽게 이관합니다.
  - **🛑 프로세스 중단**: 동작 중인 셀레늄 크롬 웹 드라이버를 즉각 파괴하고(`driver.quit()`), 크롤러 스레드 루프 내의 동작 감지 폴링을 인터럽트하여 즉시 기동을 강제 종료시킵니다.

## 7. ⚙️ 대시보드 전체 UI 설정 자동 보존 (Auto-Save & Load)

사용자가 설정값을 변경할 때마다 대시보드가 자동으로 이를 영구 저장하여 앱 재실행 시 원래 상태를 복원하도록 설계되었습니다.

### 7.1. 자동 저장 대상 옵션 (총 20개)
- **플레이보드 크롤러 탭**: 수집 대상, 일괄 크롤링 여부, 카테고리, 국가, 기간, 특정 날짜 수집 여부, 날짜 선택값, 로그인 모드 여부, 수집 제한 개수, 수집 기준 (조회수/좋아요/댓글 순위)
- **DB 통계 및 검색 탭**: 검색 키워드, 데이터 소스 필터, 컨텐츠 타입 필터, 최소 조회수, 최대 조회수, 검색 게시일 시작일, 검색 게시일 종료일, 정렬 기준
- **API 동기화 탭**: API 동기화 채널 URL, 최신 영상 수집 한도 개수

### 7.2. 직렬화 및 스마트 쓰기
* **백업 경로**: `output/settings.json`
* **타입 변환**: `st.date_input` 등에서 사용되는 `datetime.date` 객체는 JSON에서 지원하지 않는 형식이므로, 로컬 파일 쓰기 시에는 문자열(`YYYY-MM-DD`)로 자동 직렬화하고, 로드 시에는 `datetime.date` 인스턴스로 자동 파싱하여 각 위젯에 온전히 주입합니다.
* **스마트 저장 필터**: 매번 위젯을 바꿀 때마다 JSON 쓰기 I/O를 빈번하게 실행하여 시스템 부하를 일으키는 것을 막기 위해, 세션 상태에 마지막으로 저장된 스냅샷 딕셔너리와 대조하여 **실제 설정값의 변경이 감지되었을 때만** JSON 파일을 자동 갱신하도록 설계되었습니다.

## 8. 📊 수집 기준(조회수/좋아요/댓글 순위) 자동 탭 전환 정책

사용자가 수집 기준을 변경하여 수집을 개시할 때 셀레늄이 자동으로 브라우저 내 해당 정렬 차트 탭으로 전환하여 크롤링합니다.

### 8.1. UI 탭 버튼 식별 및 자동 전환 클릭
* **동작 시점**: `driver.get(url)`로 첫 페이지 접속을 완료한 뒤, 만약 로그인 모드가 활성화되어 있다면 사용자가 구글 로그인을 완전히 마친 이후(수동 재개 신호 혹은 로그인 감지 이후)에 실행됩니다.
* **요소 식별 기술**: Playboard의 랭킹 분류 탭의 구조적 마크업(`<li class="item"> <span class="item__label">조회수/좋아요/댓글 순위</span> </li>`)을 타겟팅합니다. 텍스트에 앞뒤 공백이 존재할 수 있으므로, XPathcontains 함수(`//li[contains(@class, 'item')]//span[contains(text(), '{ranking_criteria}')]`)를 사용하여 대상 탭을 탐색합니다.
* **JS 클릭 및 로드 지연 보장**: 브라우저 오버레이 또는 로그인 팝업으로 인해 클릭 간섭이 발생하는 것을 방지하기 위해 셀레늄의 `click()` 대신 Javascript API(`arguments[0].click();`)로 다이렉트 클릭을 지시하며, 탭 클릭 직후 서버와 DOM 렌더링 엔진이 데이터를 리로드할 수 있도록 `2.0` ~ `3.0`초의 충분한 지연 시간을 보장합니다.
* **예외 처리 정책**: 채널(channel) 랭킹의 경우 해당 조회수/좋아요/댓글 탭 자체가 존재하지 않으므로, `target_type != 'channel'`인 경우에만 탭 전환 로직이 구동되며, 요소를 찾지 못할 경우에는 에러로 멈추지 않고 Warning 로그를 발생시킨 뒤 기본값(조회수 순위)으로 수집이 흘러가도록 예외 처리를 정교하게 구축했습니다.

## 9. 🔄 브라우저 세션 재사용 및 중복 수집 방지 (이어서 수집)

수집 효율성을 극대화하고 네트워크 리소스를 절약하기 위해 브라우저 세션 재사용 및 이어받기 메커니즘을 도입했습니다.

### 9.1. 브라우저 세션 재사용 (Keep-Alive)
- **동작**: 일괄 크롤링(Batch Mode)을 구동할 때 카테고리 전환 마다 브라우저를 닫고 새로 켜는 무거운 기동 과정을 폐지했습니다.
- **원리**: 크롤러에 `keep_open=True` 옵션을 주어 활성화된 드라이버를 유지하고, 다음 카테고리 주소로 바로 `driver.get(url)` 이동을 수행하여 하나의 창에서 모든 크롤링 프로세스가 종료되도록 제어합니다.
- **클린업**: 전체 일괄 수집이 종료되거나 오류로 중단되는 시점의 `finally` 블록에서 `crawler.close()`를 실행해 브라우저를 완벽하게 종료합니다.

### 9.2. 지정 날짜(과거 특정 날짜 및 오늘 자) 중간 파일 이어서 수집 및 최적화 순환 (중복 수집 방지)
- **동작**: 수집 중 오류나 사용자의 강제 중단이 발생한 뒤 다시 기동할 때, 또는 과거 특정 날짜의 데이터를 다시 수집하려 할 때, 대상 날짜의 수집 디렉토리(`output/<date_folder>/<type_folder>`)에 이미 매칭되는 수집 중간 파일(CSV)이 감지되면 신규 생성 대신 이어서 수집을 개시합니다.
- **원리**:
  1. 과거 특정 날짜를 선택한 경우, 동적으로 해당 날짜(예: `2026_06_09`)의 하위 디렉토리를 탐색하도록 `find_existing_batch_file`을 연동하며, 파일 저장 시에도 `generate_safe_filepath`에 `ranking_date`를 넘겨 동일한 과거 날짜 디렉토리에 정확히 일치하여 생성/업데이트되도록 구현되었습니다.
  2. 수집 기준(Criteria)까지 파일명에 결합하여 수집 조건이 완벽히 동일한 기존 파일을 찾습니다.
  3. 기존 파일의 데이터 행 수(예: 40개)를 파악하고, 웹페이지 상에서는 목표 개수(Limit)만큼 스크롤을 시도합니다.
  4. 실제 HTML 파싱 단계에서는 **기존 행 개수(start_rank) 이후의 랭킹 요소들만 파싱**하여 속도와 리소스를 대폭 아낍니다.
  5. 크롤링 완료 후 기존 데이터프레임과 새로 수집된 데이터프레임을 Concat 병합하고, 파이썬 기동 시 변경되는 동적 해시 비디오 ID 문제를 원천 해결하기 위해 쇼츠/비디오는 `["title", "channel_name"]` (제목 + 채널명) 조합을, 채널은 `["channel_name"]` (채널명) 조합을 기준으로 중복을 정밀 필터링(drop_duplicates)합니다.
  6. 중복 제거로 인해 발생할 수 있는 순위(Rank) 누락 및 꼬임 현상을 막기 위해, 데이터프레임을 재정렬한 후 1위부터 순차적 순번을 다시 정상 재부여합니다.
- **일괄 크롤링 순환 최적화**:
  - 일괄 크롤링 시, 시작 시점에 12개 카테고리의 기존 파일들을 미리 전수조사하여 이미 목표 개수(Limit)를 완벽히 채운 카테고리는 **브라우저 실제 순회 루프(`needs_crawl`)에서 완전히 배제**합니다.
  - 제외된 카테고리의 데이터는 루프 시작 전 디스크에서 즉시 로드하여 최종 병합 버퍼(`all_data`)에 적재해두며, 부족한 수량이 있는 카테고리들만 실제로 브라우저를 열어 크롤링을 동작시킴으로써 전체 처리 속도와 자원 소모를 극적으로 줄였습니다.
- **기존 파일 업데이트**: 새로운 타임스탬프 파일을 생성하지 않고, **감지된 기존 파일의 경로 그대로 데이터를 덮어써서 갱신**합니다.
- **데이터프레임 타입 자동 표준화 정합성 보정 (Type Casting)**:
  - 수집 이어서 받기 또는 병합(`pd.concat`) 진행 시 발생할 수 있는 데이터 구조 및 자료형의 불일치 문제를 해결하기 위해 `standardize_dataframe_types` 프로세스를 강제 적용합니다.
  - `Rank`, `Views`, `Likes`, `Comments` 등의 수치형 컬럼은 결측치를 `0`으로 처리하고 모두 `int64` 정수 타입으로 통일합니다.
  - `Video ID`, `Video Title`, `Channel Name`, `Subscribers`, `Upload Date` 등의 문자열 컬럼은 결측치를 `'N/A'`로 치환한 뒤 `str` 타입으로 통일하고 앞뒤 공백을 자동으로 다듬어 정제합니다.
  - 이를 통해 기존 데이터와 신규 수집 데이터 간의 정렬(`sort_values`) 혹은 연산 과정에서 발생할 수 있는 `TypeError`를 원천 차단하고 완벽한 데이터 품질을 보장합니다.

## 10. 📂 수집 데이터 CSV 컬럼 구성

저장되는 CSV 파일의 구조는 후속 자막 추출기 연동과 데이터 식별이 용이하도록 다음과 같이 정렬 및 구성됩니다.

* **컬럼 정렬 순서**:
  1. `Period` (수집 기간 - 일간/주간/월간)
  2. `Ranking Date` (랭킹 기준 날짜)
  3. `Type` (콘텐츠 유형 - shorts/video/channel)
  4. `Country` (수집 대상 국가)
  5. **`Category`** (수집 대상 카테고리 - 사용자 요청에 의해 Country 오른쪽에 배치)
  6. **`Criteria`** (수집 정렬 기준 - 조회수/좋아요/댓글 순위 등, 사용자 요청에 의해 Country 오른쪽에 배치)
  7. `Rank` (순위)
  8. `Rank Change` (순위 변동값)
  9. `Video Title` (영상 제목)
  10. `Views` (조회수)
  11. `Upload Date` (업로드 날짜)
  12. `Tags` (태그 문자열)
  13. `Channel Name` (채널명)
  14. `Subscribers` (구독자 수)
  15. `Thumbnail` (썸네일 URL)
  16. `Video ID` (자막 추출용 비디오 고유 ID - NameError 수정 및 컬럼 포함 보완)

## 12. 🔍 크롤링 동작 예측 상태 표시 기능 (Pre-Crawl Predictor)
크롤링을 실행하기 전에 현재 설정한 파라미터(수집 대상, 카테고리, 국가, 기간, 정렬 기준 및 목표 개수)를 기반으로 오늘 자 수집 현황을 실시간 분석하여 예측 동작 상태를 표시합니다.

* **단일 카테고리 크롤링 예측**:
  - **[신규 파일 생성]**: 오늘 자 파일이 없는 경우, 새롭게 목표 개수만큼 크롤링함을 안내합니다.
  - **[기존 파일 채우기]**: 이미 수집된 개수가 목표 수량보다 적을 경우, 부족한 잔여 개수를 추가 수집하여 기존 파일을 보완/업데이트함을 표시합니다.
  - **[수집 완료 건너뜀]**: 오늘 자 수집된 개수가 이미 목표 수량을 충족한 경우, 중복 리소스를 낭비하지 않도록 크롤링을 건너뜀을 안내합니다.
* **일괄 카테고리 크롤링 예측**:
  - 세부 모든 카테고리를 대상으로 오늘 자 수집 현황을 일괄 대조하여 요약 요소를 표시합니다 (예: `신규 생성: X개, 기존 채우기: Y개, 건너뜀: Z개`).
  - 사용자는 `🔍 카테고리별 상세 예측 현황 보기` 접기창(Expander)을 확장하여 각 세부 카테고리별 예측 동작과 목표 잔량을 한눈에 사전에 파악할 수 있습니다.
  - **클립보드 복사 지원**: 예측 테이블 아래에 렌더링된 텍스트 상자 우측 상단의 **복사(📋) 아이콘**을 누르면, Streamlit의 강력한 내장 복사 컴포넌트인 `st.code`를 통해 브라우저 보안 제약이나 iframe 격리 여부와 무관하게 엑셀이나 구글 스프레드시트 등에 곧바로 붙여넣기(Ctrl+V)를 할 수 있는 탭 구분(TSV) 텍스트 데이터 형식으로 클립보드에 즉시 복사됩니다.
* **예측 상태 실시간 자동/수동 갱신**:
  - **수동 실시간 갱신**: 사용자가 언제든지 디스크 파일을 재스캔하여 예측 정보를 갱신할 수 있도록 `🔄 예측 상태 실시간 갱신` 버튼을 제공합니다. 버튼을 클릭하면 세션 버퍼에 저장된 이전 크롤링 흔적을 모두 지우고 완전히 깨끗하게 디스크 파일들을 재조회하여 예측 테이블을 갱신합니다.
  - **크롤링 완료 후 자동 갱신**: 크롤링 작업이 성공적으로 끝나거나 혹은 예외/중단으로 종료될 때, 스크립트 실행 마지막 단계(`finally:` 블록)에서 `st.rerun()`을 자동으로 강제 호출합니다. 이로 인해 크롤링이 완료되는 즉시 사용자가 별도로 웹페이지 새로고침이나 갱신 버튼을 누르지 않아도 좌측의 `🔍 동작 예측 상태` 패널이 디스크의 실제 수집 상황을 자동으로 반영하여 최신화됩니다.
  - **세션 기반 결과 프리뷰 보존**: `st.rerun()` 호출로 인해 우측 "실시간 크롤링 현황판"의 수집 완료 데이터와 파일 다운로드 버튼이 소실되는 현상을 막기 위해, 크롤링 성과 데이터를 `st.session_state['crawl_result']` 버퍼에 임시 저장하여 새로고침 이후에도 데이터프레임 프리뷰와 CSV 다운로드 기능이 화면에 지속적으로 남아 있도록 안정성을 확보했습니다.

## 13. 📅 사이드바 최근 7일 수집 현황 요약 (Sidebar History Monitor)
사이드바의 `📊 최근 7일 Quota 추이` 확장 패널 하단에 `📅 최근 7일 수집 현황` 섹션이 상시 배치됩니다.

* **동작 원리**:
  - 오늘을 기준으로 최근 7일간의 날짜 디렉토리(`output/YYYY_MM_DD/`)에 대상 수집 타입(Shorts/Video/Channel 등)의 파일들이 실제 존재하는지 실시간으로 스캔합니다.
  - 파일이 존재하여 수집 이력이 감지된 날짜는 자동으로 개별 접기창(Expander)으로 동적 리스팅됩니다. (오늘 자는 기본 활성화 상태)
  - 각 날짜의 접기창 내부에는 해당 설정 조건(국가, 기간, 랭킹기준 등)으로 실제 수집량이 존재하는 카테고리들만 추출하여 `카테고리`, `수집량`, `부족분(목표 제한 개수 대비 잔량)` 테이블을 직관적으로 시각화합니다.
  - 테이블 하단에는 수집 수량이 목표치에 도달하지 못한 카테고리 목록(⚠️ 미달)을 캡션 메시지로 노출해 한눈에 일자별 크롤링 완성도를 모니터링할 수 있도록 돕습니다.

## 14. 🏆 최종 수집 완료 통계 리포팅 정책 (Crawl Summary Reporter)
크롤링 수집 작업이 성공하거나 혹은 예외 등으로 인해 중단/실패한 경우, 수집 전반의 완성도와 수치를 종합적으로 검증할 수 있도록 로그와 대시보드 화면상에 통계 리포트를 강제 표시합니다.

* **통계 집계 항목 (stats)**:
  - `목표 카테고리 수`: 단일 크롤링의 경우 1개, 일괄 크롤링의 경우 전체 서브 카테고리 수 (12개)
  - `카테고리당 목표량`: 사용자가 대시보드에서 지정한 수집 목표 제한 개수(Limit)
  - `건너뛴 카테고리 수` (일괄): 이미 목표치에 달성되어 실제 웹 브라우저 접속을 건너뛴 카테고리 수
  - `실제 업데이트된 파일 수`: 이번 실행에서 신규 수집 또는 기존 파일 채우기가 동작하여 디스크 파일이 새로 써진 개수
  - `업데이트된 파일의 총 행 수`: 실제 업데이트가 진행된 파일들에 수집 완료되어 최종 보존된 총 영상 데이터 행 수
  - `새로 추가 수집된 영상 수`: 이번 실행 중에 실제로 웹 페이지로부터 새로 파싱되어 유입된 순수 신규 영상 수
  - `수집 실패 카테고리 수 및 목록`: 수집 동작 도중 오류가 발생한 카테고리의 개수와 구체적인 카테고리 이름
* **로깅 및 결과창 출력 포맷**:
  - **터미널 로깅**: 프로세스 완료 직후 `logger.info`를 활용하여 `🏆 [일괄/단일 크롤링 최종 수집 통계 요약 (성공/실패)]` 형식의 박스형 요약을 터미널 콘솔 및 로그 파일에 상세히 인쇄합니다.
  - **대시보드 시각화**: `st.session_state['crawl_result']` 결과창 최상단 영역에 `st.columns`를 활용한 4열 메트릭 카드(목표 카테고리 수, 목표량, 업데이트 파일 수, 실제 업데이트 행 수)를 렌더링하고, 새로 수집된 건수를 캡션으로 표시하며, 실패한 카테고리가 존재할 시 빨간색 에러 박스(`st.error`)를 통해 그 목록을 직관적으로 시각화합니다.

## 15. 📅 실질 랭킹 날짜 자동 감지 선택 및 네이밍 정합성 보장 (Date Selector & Naming Policy)

플레이보드는 당일에 실시간 랭킹 데이터 업데이트가 늦어질 수 있어(예: 11일 기동 시점에 리스트 상의 최신 데이터가 10일 자인 경우 등), 수집 시점의 오늘 날짜가 아닌 **실질적인 랭킹 데이터 날짜**를 자동으로 감지/선택하고 이를 바탕으로 저장 파일과 폴더의 경로명을 네이밍합니다.

* **동작 시점**:
  - 첫 랭킹 페이지 로드 및 로그인 처리가 완료된 시점의 수집 기준 탭 전환 직후와, 수집 도중 로딩 지연 등으로 브라우저가 새로고침(F5)되는 복구 단계 직후에 연동하여 실행됩니다.
* **날짜 리스트 자동 선택 및 실제 랭킹 날짜 파싱**:
  - **과거 특정 날짜 수집 활성화 시 (`use_specific_date=True`)**: 사용자가 지정한 `ranking_date` (예: `2026-06-11`)를 기반으로 **1일 전 날짜**를 자동 계산하여 타겟 날짜(예: `2026.06.10`)가 포함된 요소를 찾아 클릭합니다.
  - **기본 최신 날짜 수집 시 (`use_specific_date=False`)**: 사이트 지연으로 최신 데이터 날짜가 늦는 현상을 방어하기 위해, 브라우저에 표시된 날짜 리스트 중 **가장 첫 번째 항목(가장 최신 날짜)**을 강제 선택 클릭합니다.
  - **날짜 라벨 파싱**: 클릭된 날짜 라벨(예: ` 2026.06.10.(수)`)에서 `YYYY-MM-DD` 형태의 실제 데이터 랭킹 날짜를 정규식으로 파싱하여 `self.actual_ranking_date` 변수에 담고, 데이터프레임의 `'Ranking Date'` 컬럼을 이 값으로 보정해 채웁니다.
* **실제 데이터 날짜 기반 네이밍 정책 (중요)**:
  - 수집을 마친 최종 데이터프레임에서 실제 감지된 랭킹 날짜(예: `2026-06-10`)를 추출하여, 이를 바탕으로 로컬 output 폴더 경로와 CSV 파일명을 네이밍합니다 (`output/2026_06_10/Shorts/` 하위에 저장).
  - 대시보드의 크롤링 동작 예측 상태 및 기존 파일 이어서 받기 프로세스에서도 대시보드 선택일의 1일 전 날짜를 미리 예측일로 보정해 기존 파일을 올바르게 매칭시켜서 이어서 수집이 오류 없이 흘러가도록 보장합니다.

## 17. 📅 수집 날짜의 실질 데이터 기준 동기화 및 네이밍 규칙 개선

수집 시스템 시간과 실제 랭킹 사이트의 기준일 불일치 문제를 해결하기 위해, 모든 대시보드와 파일 관리 규칙을 실질 데이터 기준으로 일치시켰습니다.

* **데이터베이스 `crawled_at` 필드와 `Ranking Date` 동기화**:
  - 기존에는 DB 적재 시 단순히 시스템 구동 시간(`datetime.now()`)을 기록해 대시보드 날짜와 파일 날짜가 어긋나는 혼선이 있었습니다.
  - 수집된 데이터프레임 내부의 `Ranking Date` (예: `2026-06-10`) 컬럼 값을 추출 및 파싱하여 DB 테이블의 `crawled_at` 필드로 덮어쓰도록 삽입 로직을 변경했습니다.
  - 이에 따라, 대시보드의 `수집 날짜 선택` 드롭다운과 사이드바의 `📅 최근 7일 수집 현황`에 표기되는 모든 날짜 목록들이 시스템 수집 시간이 아닌 실제 데이터의 랭킹 기준일로 완벽하게 동기화되어 표기됩니다.
* **파일명 및 폴더명 시간초 제거 정책**:
  - CSV 결과 파일 생성 시 파일명 및 저장 폴더명 꼬리에 붙던 `_시_분_초` 형태의 시간 단위(`_14_33_03`)를 완전히 배제했습니다.
  - 모든 파일과 폴더는 실제 감지된 랭킹 날짜 문자열(`YYYY_MM_DD` 형태)만을 식별자로 활용하여 `shorts_ALL_한국_일간_조회수 순위_2026_06_10.csv` 처럼 생성되므로, 꼬리표가 중복으로 여러 개 분산 생성되지 않고 랭킹일자 기준으로 덮어쓰기 형태로 안전하게 수집 이력이 갱신 및 유지됩니다.

## 18. 🧭 에러 페이지(404) 우회 및 UI 필터 자동 세팅 정책

플레이보드는 날짜 기준 파라미터(`?period=xxxx`)가 만료되었거나 미래 날짜와 같이 데이터가 아직 빌드되지 않은 주소로 접속을 시도할 경우 "페이지를 찾을 수 없습니다" 404 오류 화면을 표출합니다. 이를 안정적으로 우회하고 수집을 보장하기 위해 다음과 같은 지능형 우회 제어 메커니즘을 적용했습니다.

### 18.1. 404 에러 페이지 감지
* **동작 시점**: `driver.get(url)`로 타겟 주소에 첫 진입한 직후, 약 3초간의 페이지 안정화 과정을 거친 후에 실행됩니다.
* **식별 기준**: 화면상에 에러 안내용 컨테이너(`div.common-error`)가 발견되거나, "페이지를 찾을 수 없습니다" 에러 문구, 또는 "홈 화면으로 가기" 버튼(클래스 `tbtn--black` 등)이 표시되는지를 DOM 스캔을 통해 탐지합니다.

### 18.2. 기본 루트 URL 우회 및 UI 필터링 자동 세팅
* **기본 루트 URL로 재접속**: 404 에러 상태가 감지되면, 크롤러는 강제 정지하는 대신 수집하고자 하는 대상 유형(`target_type`)의 기본 루트 주소로 즉시 우회하여 재접속을 수행합니다.
  - `shorts` (쇼츠) -> `https://playboard.co/chart/short/`
  - `video` (일반 영상) -> `https://playboard.co/chart/video/`
  - `channel` (채널) -> `https://playboard.co/youtube-ranking/`
* **UI 필터 직접 제어 (`_apply_ui_filters`)**: 기본 주소로 진입하면 브라우저 상의 세팅이 초기화되어 있으므로, 크롤러는 대시보드에서 지정했던 수집 파라미터들(**국가, 기간, 카테고리**)을 브라우저 화면에서 차례대로 직접 찾아 클릭합니다.
  - **선택 상태 판별**: 불필요한 새로고침을 방지하기 위해 각 리스트 항목(`li.item`)에 `item--selected` 클래스가 붙어 이미 선택된 상태인지 먼저 확인합니다.
  - **검색 폴백 (Fallback Search) 및 슬래시(/) 카테고리 검색 보완**: 카테고리나 국가의 가짓수가 많아 화면상에 즉시 클릭 가능한 항목이 보이지 않을 경우, 각 영역의 검색창(`input.search__input`)에 설정된 텍스트를 직접 입력(`send_keys`)하고 엔터를 수행하여 필터 검색 후 선택하는 방어 코드를 내장해 완벽한 클릭을 보장합니다. 특히, `인물/블로그`, `뉴스/정치` 등 슬래시(`/`)가 포함된 카테고리의 경우, 플레이보드 검색 필터의 한계로 인해 전체 명칭을 검색창에 입력하면 결과가 노출되지 않는 문제가 있었습니다. 이를 우회하기 위해 슬래시 이전의 첫 번째 단어(예: `"인물"`)만 검색어로 전송해 결과를 노출시킨 뒤, 목록에서 전체 카테고리 명칭(`normalize-space()='인물/블로그'`)과 일치하는 항목을 XPath로 매칭하여 정상 선택하도록 보완했습니다.
  - **날짜 및 기준 클릭 연계**: 국가/기간/카테고리 세팅이 완료되면 기존의 정렬 기준 탭 전환(`_switch_criteria_tab`)과 랭킹일자 선택(`_select_ranking_date`)을 순차적으로 호출하여, 최종적으로 404 에러가 나지 않는 최신 날짜의 올바른 타겟 차트 페이지로 브라우저를 강제 이동시킨 후 안전하게 무한 스크롤 수집을 시작합니다.
* **최적화 로직**: `use_specific_date` (과거 특정 날짜 랭킹 수집) 체크박스가 **활성화되어 있을 때만** `timestamp`를 연산하여 URL에 삽입하도록 구조를 개선했습니다. 최신 날짜 수집 모드일 때는 `timestamp`를 `None`으로 설정해 404 에러가 없는 순수 종합 차트 주소(예: `https://playboard.co/chart/short/most-viewed-all-videos-in-south-korea-daily`)로 즉시 직행합니다. 이를 통해 우회 대기 시간(3~4초) 소모 없이 즉각적으로 최신 날짜 수집을 성공시킬 수 있도록 최적화했습니다.

## 19. 🐛 수집 기준별 크롤링 예외 및 KeyError 방지 로직 (KeyError: 'Views')

### 19.1. 예외 원인 및 설계 개선
* **원인**: 플레이보드에서 "댓글 순위"나 "좋아요 순위" 탭을 기준으로 크롤링을 수행할 경우, 수집된 지표를 저장하는 `item_dict` 생성 과정에서 해당하는 수집 기준의 메트릭(`Comments` 또는 `Likes`)만 딕셔너리에 담기게 되어 `'Views'` 컬럼 키가 누락되었습니다. 이후 크롤러의 데이터 유효성 검증(`valid_items`) 및 로그 출력 부에서 `'Views'` 키에 직접 접근을 시도해 `KeyError`가 발생해 수집 루프가 전면 중단되는 문제가 존재했습니다.
* **방어 대책**:
  - `PlayboardCrawler._parse_videos`에서 `item_dict`를 생성할 때, 수집 기준에 무관하게 **`Views`, `Likes`, `Comments` 3대 수치 컬럼을 모두 `0`으로 기본 초기화**한 뒤 실제 파싱된 데이터를 덮어씌우도록 개선했습니다.
  - 디버그 요약 로그 출력 시 현재 수집 기준(`metric_col`)에 맞춰 지표명을 유연하게 출력하도록 개선하고, 수집 정합성을 검사하는 `valid_items` 추출부 역시 하드코딩된 `Views` 대신 현재의 `metric_col`을 사용하여 유효 레코드를 집계하도록 보완했습니다.

## 20. 📋 Streamlit 대시보드 복사 팝업 모달 설계 및 Iframe Sandbox 우회 패턴

Streamlit은 본질적으로 다중 Iframe 구조 내에서 실행되며, 일반적인 HTML 삽입 방식(`st.html` 또는 마크다운) 내부에 `<script>` 태그를 두면 React 가상 DOM의 렌더링 생명주기에 의해 해당 코드가 동작하지 않거나 제거됩니다. 대시보드 카드 클릭 시 노출되는 복사 팝업 모달은 이를 극복하기 위해 다음과 같이 특수 설계되었습니다.

### 20.1. 투명 컴포넌트 (`components.html`) 및 부모 DOM 우회 바인딩
* **투명 컴포넌트**: `streamlit.components.v1.html(..., height=0)` 모듈을 호출하여 화면 상에 영역을 차지하지 않는 투명한 Iframe 하부 컴포넌트를 강제 렌더링하고, 이 컴포넌트 내부에서 부모 페이지의 DOM 객체인 `window.parent.document`를 참조하여 이벤트를 위임합니다.
* **이벤트 리스너 중복 바인딩 및 누수 차단**:
  - Streamlit은 조작이 가해질 때마다 전체 페이지 스크립트가 반복 실행(리런)되므로, `window.parent`에 단순 이벤트 리스너를 추가하면 매 리런마다 동일한 리스너가 누적 등록되어 팝업이 다중 트리거되는 현상이 유발됩니다.
  - 이를 방지하기 위해 `window.parent.copyModalWired = true` 전역 플래그를 두어 최초 1회만 이벤트 리스너가 바인딩되도록 통제했습니다.
* **이벤트 위임 (Event Delegation)을 통한 카드 연동**:
  - 부모 DOM 레벨에서 전체 `click` 이벤트를 감청하고, `e.target.closest('.dashboard-card, .list-card')` 구문을 활용해 동적으로 추가/갱신되는 카드의 클릭 여부를 판단합니다.
  - 카드가 클릭되면 카드 내부의 `.card-title`, `.card-channel-name`, `.list-title`, `.list-channel-name` 요소를 쿼리하여 복사 모달 내부에 세팅하고 모달의 투명도(`opacity`) 및 디스플레이 속성을 토글(Transition 적용)하여 띄워줍니다.

### 20.2. 텍스트 복사 폴백(Fallback) 및 정제 메커니즘
* **클립보드 API 폴백**: 최신 `navigator.clipboard.writeText` API를 활용하여 복사를 진행하되, 보안 컨텍스트(HTTPS가 아니거나 로컬의 특정 격리 환경)로 인해 클립보드 API가 거부당할 때를 대비해 `document.execCommand('copy')` 임시 텍스트 영역(textarea) 생성 방식을 폴백으로 내장하여 100% 신뢰성 있는 복사를 실현했습니다.
* **복사 텍스트 정제**: 리스트형 카드 렌더링 시 UI 가독성을 위해 추가되었던 `'👤 채널명:'` 및 기타 이모지 텍스트들이 복사 텍스트에 섞이지 않도록, 스크립트 레벨에서 정규식 치환(`.replace('👤 채널명:', '').trim()`)을 적용하여 **사용자가 원하는 순수한 영상 제목과 채널명만** 클립보드에 적재되도록 보정 로직을 수립했습니다.

## 21. 🔄 다중 수집 기준(Criteria) 데이터 병합 및 정합성 보정

### 21.1. 동일 영상 중복 누적 문제 및 해결
* **문제 배경**: 동일한 날짜, 국가, 기간 내에서 '조회수 순위', '댓글 순위' 등 다른 수집 기준으로 수집된 데이터는 DB 테이블(`shorts_rank`, `videos_rank`, `channels_rank`)에 중복된 비디오 정보로 적재됩니다.
  - 기존 대시보드 로직은 중복을 회피하기 위해 `drop_duplicates`를 수행하여 단순히 하나만 보존하고 나머지는 삭제했습니다. 이로 인해 '조회수' 기준으로 수집된 행이 소실되면 대시보드에 댓글수만 표시되고 조회수는 `0`으로 나타나는 등의 메트릭 정보 손실 문제가 발생했습니다.
* **해결 로직 (Pandas GroupBy & Max Aggregation)**:
  1. **고유 병합 식별자(merge_key) 정의**:
     * 쇼츠/일반 영상: 고유한 `video_id`가 `'N/A'` 또는 `'None'`이 아닐 때 이를 최우선 매칭 키로 지정하고, 누락 시에는 `'title'` + `'channel_name'`의 결합키를 생성해 식별합니다.
     * 채널: `'channel_name'`을 고유 식별 키로 사용합니다.
  2. **수치형 컬럼 정합성 통일**: 병합 연산 과정의 정밀도를 보장하기 위해 `views`, `likes`, `comments` (채널은 `total_views`, `subscriber_count`) 등의 수치형 컬럼을 정수타입(`int64`)으로 강제 캐스팅하고 결측치를 `0`으로 전처리합니다.
  3. **메타데이터 및 수치 지표 결합**:
     * 전체 데이터프레임을 `crawled_at_parsed` 내림차순(가장 최근 수집본) 및 `rank` 오름차순(가장 높은 랭킹)으로 정렬해 대표 행이 `keep='first'`에 보존되도록 준비합니다.
     * `groupby('merge_key')`를 수행하여 그룹 내의 수치형 컬럼 최대값(`max()`) 데이터를 별도로 추출합니다.
     * `drop_duplicates(subset=['merge_key'], keep='first')`를 실행해 메타데이터가 완벽히 보존된 1건의 대표 행만 남긴 뒤, 기존 수치 컬럼을 제거하고 추출해둔 그룹 내 최대 수치 데이터와 `pd.merge`하여 병합합니다.
  4. **비율 연산 재적용**: 병합 처리가 완료된 뒤, 최종 합산된 조회수 및 구독자수 정보를 바탕으로 `view_sub_ratio`를 새로 연산하여 갱신합니다. 이를 통해 지표 왜곡을 완벽히 방어합니다.

## 22. 📋 모든 수집기준 일괄 순회 수집 엔진

단 한번의 크롤러 기동만으로 플레이보드의 3대 정렬 차트(조회수 순위, 좋아요 순위, 댓글 순위)를 모두 자동 수집하도록 설계된 2중 순방향 크롤링 엔진입니다.

### 22.1. UI 옵션 제어 및 다차원 상태 예측
* **체크박스 및 셀렉트박스 상호작용**:
  * 사용자가 `📋 모든 수집기준 수집하기` 체크박스를 활성화하면, 기존의 단일 정렬 `수집 기준 (Criteria)` 셀렉트박스는 비활성화(`disabled=True`) 처리되어 오작동을 방지합니다.
* **Pre-Crawl Predictor(동작 예측 상태)의 다차원 확장**:
  * 체크박스 활성화 시, 탐색 대상 기준이 `[조회수 순위, 좋아요 순위, 댓글 순위]` 3개로 확장됩니다.
  * 단일 카테고리 모드 시에는 3개 기준 각각에 대해, 일괄 카테고리 모드 시에는 `3대 기준 x 12개 카테고리 = 총 36회` 크롤링 태스크에 대해 디스크 상의 CSV 파일 존재 여부를 스캔합니다.
  * 각 조건에 대한 기존 수집량(이미 채워진 데이터 개수)을 판별하여 `📂 신규 생성`, `🔄 기존 파일 채우기`, `✓ 완료 건너뜀` 상태의 상세 예측 요약 표를 동적으로 렌더링하고 복사용 TSV를 출력합니다.

### 22.2. 브라우저 세션 재사용 및 예외 자원 클린업
* **Selenium Chrome 브라우저 세션 재사용 (`keep_open=True`)**:
  * 여러 개의 차트나 카테고리를 순회 수집할 때 매 태스크마다 웹 드라이버를 생성 및 파괴하게 되면 성능 낭비와 메모리 누수가 극심해집니다.
  * 이를 극복하기 위해 `PlayboardCrawler` 인스턴스의 브라우저 세션을 실행 루프 전반에서 재사용(`keep_open=True`)합니다. 전환이 발생하더라도 동일한 브라우저 창에서 URL 및 필터 클릭만 전개하여 자원 사용률을 혁신적으로 절감합니다.
* **안전한 세션 파괴 및 메모리 회수**:
  * 프로세스 중도 실패, 사용자 중단(`stop_requested`), 혹은 전체 작업의 정상 완료 시 `finally` 블록에서 `crawler.close()`를 일관되게 호출하여 사용 중이던 Chrome 브라우저 프로세스들을 안전하게 파괴하고 시스템 자원을 반환합니다.
